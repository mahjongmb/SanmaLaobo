## プロジェクト概要

- プロジェクト名: **MBsanma**
- 内容: **三人麻雀のブラウザ実装**。一般配信を前提とした、**ハウスルール対応の練習＋検証データ分析アプリ**。
- 想定ユーザー: 普段から三麻を打っている人。自分の卓のローカルルールに合わせて打ちたい／集めたデータからルール別の傾向を分析したい層。
- 実行環境: **ローカル実行（`file://`）** を前提に作っている。サーバー前提に切り替えない。
- 使用技術: **HTML / CSS / 素のJavaScript**（フレームワーク・ビルドツール・モジュールバンドラなし）
- 永続化: **IndexedDB**（半荘ログ）＋ **localStorage**（設定・プリセット）＋ **Supabase**（アカウント機能・クラウド同期）
- 補助サーバー: `mbsanma_bridge_B/server.js`（Node.js 製のローカル橋渡し。CPU打牌/副露を OpenAI Responses API へ問い合わせる任意機能）

### 三麻ルール前提（実装の固定部分）
- 萬子は 1m / 9m のみ（中張牌の萬子なし）
- 山 108 枚
- 赤 5p / 赤 5s あり
- 北抜きあり

### `MBSanmaRulesConfig` で切替できる主な範囲（ハウスルール）
- 半荘戦 / 東風戦
- 持ち点 / 返し点 / 連荘条件
- ウマ / クビ / トビ終了 / コールド終了 / 箱下精算
- 本場点 / 聴牌料 / ツモ損 / 100点単位 / 切り上げ満貫
- 流し倍満の有無（満貫扱い／倍満扱い）
- 採用役の個別 ON/OFF（一盃口 / 二盃口 / 三色同刻 / 三槓子 / 一気通貫 / 混全帯么九 / 純全帯么九 など）
- 三麻ローカル役・萬子ホンイツ 3 択
- 牌構成（赤・金・虹などの色付き牌）と祝儀対象牌の詳細編集

これらの設定変更は**プレイ中の挙動・点数計算・精算・ログ保存・分析集計**まで横断で効くため、変更時は影響範囲を必ず意識する。

## 作業方針

### 1. 修正範囲は最小にする
- 依頼された問題に必要なファイルだけ触る
- 推測で複数ファイルへ広げない
- 大規模リファクタは禁止
- 関連影響が確実にあるときだけ周辺を直す

### 2. まず責務を確認する
修正前に、対象ファイルが何を担うかを確認する。
ファイルの役割確認には、**`other/MBsanma_役割台帳_20260418更新_v10 (1).md`** を基準として使う。
旧台帳 `other/MBsanma_役割台帳.md` は退役予定。

### 3. 最新版を正本として扱う
- 古い説明や断片コードではなく、**現在ワークスペースにあるファイル**を正本として扱う
- 修正前に対象ファイルの実内容を読む
- 記憶で書き換えない
- 別ファイルの古い責務を流用しない

### 4. 仕様の正本
- **当面の仕様正本は「現行実装」＋「役割台帳v10」**
- 将来的に **`SPEC.md`** を起こして「三麻ルールの確定事項」と「`MBSanmaRulesConfig` の切替マトリクス」を明文化する想定（未整備）
- 同様に **`BUGS.md`** を起こして「再現手順／影響ファイル／優先度」付きで不具合を管理する想定（未整備）
- これらが存在する場合はそちらを優先する。存在しない場合は現行実装を読んで判断する
- 仕様が曖昧なときは、既存実装と依頼内容の両方を確認して最小変更で合わせる

## 回答・提案のしかた

### 1. どのファイルを触るかを明示する
最初に対象ファイル名をはっきり書くこと。

### 2. バグ修正時の説明順
簡潔に、以下の順で書く。

1. 原因
2. 修正箇所
3. 修正内容

### 3. 全文提示ルール
ユーザーが**全文提示**を求めた場合は、以下を守る。

- ファイル内容を省略しない
- 変更部分だけにしない
- コメント・空行も含めて完全な内容を出す
- 「前回と同じ部分は省略」はしない

### 4. 差分提示ルール
差分でよいと明示された場合だけ、関数単位や差分単位で出してよい。
それ以外は勝手に差分形式へしない。

## 永続化・ネットワーク連携の注意

### IndexedDB / localStorage
- **半荘ログは IndexedDB 優先**で保存する（`js/match_log.js`）
- `account` / `local` でスコープを分け、ログイン状態に応じて読み書きを切り替える
- 旧 localStorage 形式からの移行ロジックは**消さない**（既存ユーザーのログを失う）
- `getStoredLogsAsync()` / `refreshStoredLogsAsync()` 経由で**非同期前提**で読む

### Supabase 連携
- `js/supabase_client.js` がアカウント発行／読込／保存／ルール同期 RPC をまとめる
- ルール同期の運用手順とテストは `docs/cloud_rules_sync_checklist.md` を参照
- **ゲストモード（未ログイン）はクラウド呼び出しをしない**
- ログイン時は **クラウド → ローカル（上書き）**、保存ボタン時は **ローカル → クラウド**（fire-and-forget）
- 設定値をいじっただけでは同期しない設計。勝手にフックしない

### bridge サーバー（任意機能）
- `mbsanma_bridge_B/server.js` は **CPU 判断を OpenAI Responses API に問い合わせる任意の橋渡しサーバー**
- ブラウザ本体から直接読まれず、`js/cpu_api_bridge.js` 経由で呼ばれる
- **必須機能ではない**。bridge が落ちていても内部 AI へフォールバックする
- ローカル `file://` 前提の本体プロジェクトとは独立して動く

## 推奨確認手順
作業時は、基本的に以下の順で考える。

1. 依頼内容から対象責務を特定する
2. 役割台帳 (`other/MBsanma_役割台帳_20260418更新_v10 (1).md`) で対象ファイルの役割を確認する
3. 実ファイルを読む（記憶で書かない）
4. **状態変更か描画変更か**を切り分ける
5. **アプリ版か mbtrial 版か**を確認する
6. **ハウスルール設定 (`MBSanmaRulesConfig`) で挙動が変わる箇所か**を確認する
7. 影響範囲を最小にして修正する
8. 必要ならログ・結果表示・分析・成績管理への影響を確認する

## 禁止事項
- 既存コードの省略
- 既存コードの勝手な削除
- 推測で複数ファイルを書き換えること
- 仕様を勝手に変更すること
- 大規模リファクタ
- render 系で状態変更
- 会話で求められていない build tool / framework 導入
- ローカル `file://` 前提を崩す変更
- アプリ版と mbtrial 版の責務統合（別系統で運用する前提を崩さない）
- 旧 localStorage 形式移行ロジックの削除
- 検証モード／対局モードのモード判定ロジックを勝手に共通化すること

## このプロジェクトで優先する価値
1. **安定**（プレイ中に落ちない・データを失わない）
2. **責務分離**（render は描画／状態変更は進行系／保存は保存基盤）
3. **既存仕様維持**（ログ互換・保存キー互換・既存ユーザーのデータを壊さない）
4. **影響範囲最小**（最小差分）
5. **軽量**（フレームワーク・ビルド工程に頼らない）
6. **ハウスルール対応**（一般配信を見据えて、設定の自由度を素直に通す）
7. **検証データの正確さ**（分析の根拠になる集計を歪めない）
